# 멀티 에이전트 확장 계획 검토서

검토일: 2026-02-22  
검토 대상: `docs/ai-operating-system-2026-02-22/05-multi-agent-expansion-plan.md`

## Executive Summary
- 확장 방향은 타당하며, 에이전트를 업무 유형별로 분리한 구조는 재작업 감소에 유리하다.
- 핵심 보완 포인트는 `auth 계열 에이전트의 프로젝트 적합화`, `체인 길이 상한`, `결정 로그 표준화`다.

## Critical Issues
1. 인증 에이전트의 기본 가정 불일치 가능성
- 위험: 에이전트가 특정 인증 스택을 전제로 잘못된 진단을 할 수 있음.
- 조치: englishapp 인증 방식/API 라우팅 규칙을 먼저 주입하는 “사전 컨텍스트 블록” 필수화.

2. 체인 난이도 증가로 인한 운영 부하
- 위험: 작은 작업에도 과도한 검토 체인이 붙어 처리량 저하.
- 조치: 작업 등급(S/M/L/XL)별 최대 에이전트 수 제한.

3. 산출물 포맷 비표준
- 위험: 문서가 쌓여도 다음 세션에서 재활용이 어려움.
- 조치: 모든 에이전트 결과에 동일 메타데이터(작업ID, 범위, 검증명령, 다음액션) 강제.

## Missing Considerations
- 비용/시간 관점 KPI:
  - 작업당 에이전트 호출 수
  - 에이전트 체인 1회 평균 소요 시간
- 예외 처리:
  - 에이전트 결과 충돌 시 우선순위(예: plan-reviewer vs architecture-reviewer)
- 종료 기준:
  - “언제 체인을 멈추고 직접 구현/수정으로 전환할지” 명시 필요

## Recommendations
1. 작업 등급별 기본 체인 고정
- S: 1-2개 에이전트
- M: 2-3개 에이전트
- L/XL: 3-5개 에이전트

2. auth 에이전트 커스터마이즈 지침 추가
- 호출 프롬프트에 다음 고정:
  - 실제 인증 미들웨어/쿠키명
  - 테스트 가능한 라우트 목록
  - 현재 서비스 실행 방식

3. 공통 리포트 헤더 강제
- `Task`, `Scope`, `Commands`, `Findings`, `Decision`, `Next`

## Approval
- 조건부 승인
- 조건:
  1. auth 에이전트 사전 컨텍스트 템플릿 추가
  2. 작업 등급별 체인 상한 정책 확정
  3. 공통 리포트 헤더 적용
