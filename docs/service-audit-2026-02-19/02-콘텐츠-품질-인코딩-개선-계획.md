# 콘텐츠 품질/인코딩 개선 계획

## 배경

- 단어장 상세에서 의미/예문 번역이 `??`, `??? ???` 형태로 보이는 케이스를 확인했다.
- 이는 인코딩 손상, 데이터 원본 품질 저하, 렌더링 fallback 미흡 중 하나 이상이 원인일 가능성이 높다.

## 목표

1. 사용자 화면에서 `?` 반복으로 깨진 문자가 절대 노출되지 않게 한다.
2. 데이터 유입 시점에서 품질 검증을 자동화한다.
3. 문제가 생겨도 UI에서는 안전한 대체 문구를 제공한다.

## 구현 계획

### 1) 원인 분리 진단

- DB에서 깨진 문자열 패턴 탐지 쿼리 작성(`%??%`, 비정상 문자 패턴)
- 업로드/임포트 파이프라인(파일 인코딩, CSV/TSV 파싱) 점검
- API 응답 직전 직렬화 단계에서 손상 여부 확인 로그 추가

예상 수정 지점:
- `lib/` 데이터 변환 유틸
- 단어장 import 관련 API/스크립트
- `prisma/` 시드/마이그레이션 데이터

### 2) 입력 검증/정규화

- 텍스트 입력 시 UTF-8 정상 여부 검증
- 제어문자/깨짐 치환문자 필터링
- 비정상 데이터는 저장 차단 + 관리자 경고 로그

### 3) UI fallback

- 의미/예문/번역 필드가 비정상일 때:
  - 사용자용: `의미 데이터 점검 중입니다` 등 한국어 대체 문구
  - 내부 로그: 레코드 ID와 필드명 기록

## 테스트 계획

1. 깨진 문자열 fixture로 API/페이지 스냅샷 테스트 추가
2. 임포트 경로에 인코딩 혼합 파일 투입 시 차단 동작 확인
3. 정상 데이터는 기존 렌더링에 영향 없는지 회귀 확인

## 완료 기준(Definition of Done)

- 프로덕션 데이터 스캔 시 `??` 패턴 0건
- E2E에서 단어장 상세 텍스트 깨짐 재현 불가
- 새 데이터 유입 경로에 인코딩 검증 자동화 적용

## 운영 계획

- 1회성 정리 스크립트로 기존 오염 데이터 정정
- 배포 후 7일간 텍스트 품질 에러 로그 집중 모니터링
