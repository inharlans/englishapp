# 운영 관측성/크론 개선 계획

## 배경

- 과거 `Scheduled Internal Cron Jobs` 실패 알림이 발생했고, 원인 파악이 로그 한 줄(`test -n ""`)에 의존했다.
- 관리자 관측성 화면은 존재하지만, 실제 지표가 비어 있는 경우가 있어 운영 추적력이 약하다.

## 목표

1. 크론 실패 원인을 즉시 식별할 수 있도록 로그/알림 품질을 높인다.
2. 환경변수 누락, 외부 API 실패, DB 실패를 분류해서 기록한다.
3. 관리자 관측성 화면에 실사용 가능한 지표를 안정적으로 채운다.

## 구현 계획

### 1) 크론 워크플로 하드닝

- 워크플로 시작 단계에서 필수 env 검증:
  - `CRON_SECRET`, `APP_BASE_URL` 등
- 누락 시 현재 값 일부 마스킹 출력 + 명시적 에러 메시지
- `curl` 응답코드/응답 본문 일부를 실패 로그에 포함

예상 수정 지점:
- `.github/workflows/cron-jobs.yml`
- 크론 대상 API 라우트(`app/api/internal/...`)

### 2) 실패 분류 및 재시도

- 에러 코드를 `CONFIG_ERROR`, `AUTH_ERROR`, `UPSTREAM_ERROR`, `DB_ERROR`로 태깅
- 일시 실패(5xx/timeout)에는 제한적 재시도 적용
- 재시도 후 최종 실패 시 Slack/이메일 알림(선택)

### 3) 관측성 파이프라인 보강

- 관리자 `/admin` 관측성 테이블 수집 배치 주기 점검
- 요청량/오류율/지연시간 집계를 최소 24시간 유지
- "수집된 지표가 없습니다" 상태를 경고로 분류

## 테스트 계획

1. 필수 env를 의도적으로 비워 실패 메시지 품질 확인
2. 정상 env에서 수동 `workflow_dispatch` 3회 연속 성공 확인
3. 관리자 관측성 표에 24시간 내 샘플 지표 노출 확인

## 완료 기준(Definition of Done)

- 크론 실패 시 원인 분류가 로그만으로 판별 가능
- 크론 워크플로 성공률이 7일 기준 99% 이상
- 관리자 관측성 표에서 공백 상태가 상시 해소

## 롤아웃

- 1차: 로그/에러코드/가드레일 배포
- 2차: 재시도 정책 적용
- 3차: 알림 채널 연동 및 운영 온콜 문서화
