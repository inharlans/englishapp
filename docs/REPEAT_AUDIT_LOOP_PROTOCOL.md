# 서비스 반복 점검/개선 루프 프로토콜

최종 수정일: 2026-02-21

## 고정 규칙

- 결제 영역(`app/pricing`, `app/api/payments/*`, PortOne/KG 설정)은 반복 개선 루프에서 건드리지 않는다.
- 코드 푸시 이후 점검을 바로 시작하지 않는다.
- 푸시 후 반드시 4분 대기한다.
- 4분 대기 후 `https://railway.com/`에서 대상 서비스 빌드/배포가 정상 완료인지 먼저 확인한다.
- Railway 배포가 실패 또는 진행 중이면 점검을 시작하지 않고 원인 확인/재배포 후 다시 확인한다.

## 1회 반복 사이클 체크리스트

1. MCP 브라우저로 실서비스 동선 점검 (결제 제외)
2. 개선점 3~5건 도출
3. 개선 문서 작성/저장 (`docs/...`에 회차 문서)
4. 코드 반영
5. 로컬 검증(최소):
   - `npm run typecheck`
   - 변경 범위 관련 테스트
6. `README.md` 최근 변경/운영 메모 업데이트
7. 커밋/푸시
8. 4분 대기
9. Railway 빌드 성공 확인
10. 성공 확인 후 다음 점검 사이클 시작

## 회차 문서 템플릿

아래 템플릿으로 `docs/service-audit-YYYY-MM-DD-loopN/ITERATION_k.md` 작성:

```md
# k회차 점검 보고서 (YYYY-MM-DD)

## 점검 범위
- (결제 제외 명시)

## 실제 점검 동선
1. ...
2. ...

## 발견 이슈
1. [심각도] ...
2. [심각도] ...

## 반영 작업
1. ...
2. ...

## 검증 결과
- typecheck: pass/fail
- 테스트: pass/fail
- 배포 확인(Railway): success/fail

## 다음 회차 우선순위
1. ...
2. ...
```

## 중단 조건

- Railway 배포 실패가 2회 이상 연속 발생
- 로그인/권한/세션 문제로 MCP 점검 불가
- 데이터 무결성 위험이 있는 변경이 감지된 경우

중단 조건 충족 시 즉시 루프를 멈추고 원인/복구 계획을 먼저 문서화한다.
